现场问题用户隔离修复

以现场问题为权限边界，彻底解耦资源 owner 与可变化的 Claude session ID。

迭代 20260713 现有 onsite-analysis 扩展 SQLite + 本地文件 REST + WebSocket

1. 结论与设计原则

在 onsite_problems 上新增 owner_user_id，作为现场问题权限的唯一权威来源。sessions.user_id 继续服务聊天执行层，但不再参与问题列表和 REST 资源授权。

已确认风险：现有列表把 provider UUID 映射后的私有问题误判为公开孤儿；详情、消息、附件和写操作也缺少统一对象级鉴权。
核心收益：provider session ID 可以继续变化，问题 owner 始终稳定；新增路由只要使用 :id 就会自动经过统一授权守卫。
  • 服务端认证身份是唯一身份来源，不信任请求体、查询参数或 WebSocket hello 中的 userId。
  • 无 owner、磁盘孤儿、迁移冲突全部默认隐藏。
  • 越权统一返回 403，响应体不携带任何资源详情。
  • 新建问题若 owner 无法持久化，整个创建失败，不再返回“磁盘成功”。

2. 数据库与仓库层设计

2.1 Schema 变更

ALTER TABLE onsite_problems
ADD COLUMN owner_user_id INTEGER
REFERENCES users(id) ON DELETE RESTRICT;

CREATE INDEX IF NOT EXISTS idx_onsite_problems_owner_updated
ON onsite_problems(owner_user_id, updated_at DESC);

若 SQLite 版本或历史表重建逻辑不适合直接添加带外键列，则复用项目已有的“新表复制 → DROP → RENAME → 重建索引”迁移模式。

2.2 Repository API

方法职责约束
insert(problem, ownerUserId)创建问题并写入 ownerowner 必须是正整数且用户存在
listByOwner(ownerUserId)数据库先过滤 owner，再返回问题禁止扫描全量后在内存过滤
findAccessRecord(id)读取 id、owner、cwd 等授权最小字段不读取消息和附件
findOwnedById(id, ownerUserId)业务层读取已授权问题用于防止 TOCTOU
backfillOwnerByCwd()仅对唯一非空 owner 做历史回填不得覆盖已有 owner

2.3 统一授权守卫

新增 ProblemOwnershipGuard，推荐通过 Express router.param('id', ...) 覆盖所有现场问题子资源路由。守卫解析 req.user.id，读取最小授权记录并将已授权 problem 挂到 request 上。

条件响应说明
没有有效认证用户401由现有 authenticateToken 处理
问题不存在404保持现有资源不存在语义
owner 为 NULL403待认领问题不对普通用户开放
owner 与当前用户不同403统一返回 PROBLEM_FORBIDDEN
owner 匹配继续后续 handler 使用守卫加载的记录
403 响应只返回固定的 code/message，不返回实际 owner、customer、cwd、文件名、状态或消息数量。

4. 广播与并发设计

  • problems:changed 不携带问题数据，可以全局通知；客户端收到后重新调用已过滤的列表接口。
  • problem:<id>:state-changed 等资源事件必须只发给 owner 对应连接，或降级成无资源 ID 的全局刷新通知。
  • 授权通过后到写操作执行前，写 SQL 应再次带上 owner_user_id = ? 条件，避免未来转让功能引入 TOCTOU。
  • 删除、状态变更和 owner 回填应使用 SQLite transaction；文件系统删除失败时保留可重试审计信息。
  • 本方案不引入消息队列；进程内广播继续满足当前单进程部署。

5. 安全设计

访问控制

  • REST 和 WebSocket 均以服务端 token 解析结果为身份。
  • 授权集中在统一守卫，handler 不自行拼接宽松规则。
  • NULL owner、磁盘孤儿和迁移冲突全部 fail closed。
  • 敏感写操作 actor 由服务端覆盖，不信任请求体 actor_id。
  • 403 错误日志记录请求用户和 problem ID，但响应不返回 owner 信息。

文件上传与存储

  • owner 鉴权必须发生在 Multer 写临时文件之前。
  • 保留单文件 200MB、单批 20 个限制，并建议补充整批总大小限制。
  • 继续使用服务端生成的临时文件名和受控解压目录，拒绝路径穿越。
  • 不在错误响应中返回实际磁盘路径。
  • 问题目录权限至少保持当前用户专属，不扩大为组或全局可写。

输入、传输与审计

  • problem ID、cwd、状态和 reason 继续执行类型及边界校验。
  • 生产环境 REST/WSS 通过 HTTPS/WSS 传输。
  • 删除、上传、状态变更、根因确认记录 authenticated user ID、problem ID、动作和时间。
  • 审计日志不记录 token、消息正文、文件内容和密码等敏感信息。
  • 迁移报告只输出问题 ID、cwd 摘要和冲突类型，不输出客户日志内容。

7. 实施顺序与回滚

  1. 数据库：增加 nullable owner 字段、索引、回填迁移及迁移测试。
  2. 数据访问：Repository 增加 owner API，创建路径强制写 owner。
  3. 读取隔离：列表改为 owner-first 查询，删除孤儿公开逻辑。
  4. REST：接入统一守卫，先覆盖读取，再覆盖所有写操作和上传。
  5. WebSocket：删除客户端身份信任，增加 problem/cwd/owner 三项校验。
  6. 广播与审计：资源事件按 owner 定向，敏感操作使用认证 actor。
  7. 验证：迁移副本演练、双用户回归、typecheck、lint、build。
发布顺序：先部署兼容 nullable owner 的后端并完成回填，再启用严格过滤。不得先删除旧过滤逻辑、后补 owner 数据。
回滚原则：回滚应用时保留新增 owner 列和已回填数据；不要执行破坏性反向迁移。若严格过滤异常，可通过代码回滚恢复服务，但禁止恢复“NULL/孤儿公开”。
