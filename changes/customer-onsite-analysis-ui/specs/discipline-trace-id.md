# Capability: onsite.discipline.trace_id_blocked

> **Type**: ADDED
> **Source**: proposal §「纪律可视化」 + `customer-onsite-analysis/CLAUDE.md` §「traceId 必找」

## Purpose

贯彻 `customer-onsite-analysis/CLAUDE.md` 的硬约束:**traceId 全目录扫描命中 0 时,问题自动进阻塞态,禁止以"近似匹配/时间窗匹配/经验匹配"强行出根因**。该约束既约束 Claude 的输出,也约束 UI 的状态呈现。

## Requirements

### REQ-8.1 工具输出监听

The system MUST attach a streaming listener to the Claude session's tool outputs (specifically `Bash` tool results). When the listener detects a command matching `grep\s+(-r|-rc)?\s*["']?<traceId>["']?` and the tool result is an empty file list or all-zero count, the system MUST emit a `discipline:trace-id-empty` internal event within 500ms of the result arriving.

#### Scenario: Claude 跑 grep,结果全 0

- **GIVEN** 用户发了 `traceId: a1b2c3d4e5f6`
- **WHEN** Claude 调用 `Bash({ command: "grep -rc 'a1b2c3d4e5f6' pod-1/ pod-2/ iauth/" })`
- **AND** 工具返回 `{ stdout: "0\\n0\\n0\\n", exitCode: 0 }`
- **THEN** 监听器发出 `discipline:trace-id-empty`,携带 `traceId: "a1b2c3d4e5f6"` 与 `command`

#### Scenario: 不匹配的命令不触发

- **GIVEN** Claude 跑 `ls -la pod-1/`
- **WHEN** 工具返回非 grep 输出
- **THEN** 不触发 `discipline:trace-id-empty` 事件

### REQ-8.2 自动迁移到 blocked

On receiving `discipline:trace-id-empty`, the system MUST automatically:
- Call `PATCH /api/onsite/problems/:id { status: "blocked", reason: <auto-generated, see REQ-8.3> }`
- Broadcast `problem:<id>:state-changed` to all connected clients
- Inject a system message into the chat stream: a `⛔` blocked card explaining the trigger

The auto-migration MUST NOT require user confirmation and MUST NOT be cancellable.

#### Scenario: 全目录 0 命中 → 切阻塞

- **GIVEN** 问题状态 `analyzing`,用户发了 traceId
- **WHEN** 工具结果全部 0
- **THEN** 1 秒内,问题状态变为 `blocked`
- **AND** 头部徽章变琥珀
- **AND** 对话流顶部插入一张 ⛔ 阻塞卡片,内容为「traceId 全目录扫描命中 0 — 按 customer-onsite-analysis/CLAUDE.md 铁律,禁止用近似/时间窗/经验匹配强行出根因」
- **AND** 阻塞卡片包含可执行的下一步:「① 核对 traceId 是否抄全;② 补 externalweb(即 sdk-server)日志后重跑」

### REQ-8.3 自动 reason 模板

The auto-generated `reason` field MUST be at least 8 characters and MUST include: traceId, the literal command, the timestamp of the tool result, and a reference to the discipline file.

Template:
```
traceId <X> 全目录扫描命中 0 (cmd: <command>, at <ISO8601>) — 见 customer-onsite-analysis/CLAUDE.md §「traceId 必找」
```

#### Scenario: 自动 reason 完整

- **GIVEN** REQ-8.2 触发的 PATCH
- **WHEN** 写入审计
- **THEN** `onsite_state_audit.reason` 满足上述模板
- **AND** 长度 ≥ 8 字符(实际长度 ≥ 80)

### REQ-8.4 手动重跑解锁

When the user uploads additional log files (REQ-5.1) after a `blocked` state, the system MUST allow them to click "补日志后重跑分析" in the blocked card. This MUST:
- Set status back to `analyzing` with reason `用户补充日志后重跑分析`
- Send a re-prompt to Claude: `traceId <X> 仍 0 命中,以下是新上传的目录列表: <list>。请重跑全目录扫描并更新结论。`
- Log this to `onsite_state_audit`

#### Scenario: 补日志后点重跑

- **GIVEN** 状态 `blocked`,阻塞卡片显示
- **WHEN** 用户上传新日志后点「补日志后重跑分析」
- **THEN** 状态 `analyzing`,reason 含「用户补充日志后重跑分析」
- **AND** 对话流末尾追加一条 user 消息与 Claude 重跑结果

## Patch:多信号融合检测(防 C-1 误报/漏报)

> **Why**:原 REQ-8.1 只检测 `grep` 家族命令,启发式脆弱。Claude 实际可能用 `rg`/`ag`/`ack`/`find -exec grep`/`bash -c`/`cat \| grep`/MCP 工具,简单命令前缀匹配会有大量漏报。同时,Claude 一轮分析可能跑 10+ 个 grep,不是每个 0 命中都是 traceId 问题。

### REQ-8.5 多信号检测(替代 REQ-8.1 的单一命令匹配)

The system MUST detect the traceId-empty condition via **three signal sources, fused**:

| 信号来源 | 权重 | 触发动作 |
|---|---|---|
| **AI `assistant` 文本 body** 含 `/未找到\|0\s*结果\|no matches\|found nothing\|无命中\|没有结果/i` | **主信号** | emit `discipline:trace-id-empty` → 调 StateMachine 切 `blocked` |
| 工具命令匹配 `(grep\|rg\|ag\|ack) ... '<traceId>'` + stdout 全 0 | 强信号 | emit `discipline:trace-id-empty` → 调 StateMachine 切 `blocked` |
| 工具命令形如 `find ... -exec grep` / `cat ... \| grep` / `bash -c "...grep..."` + 0 命中 | 弱信号 | emit `discipline:trace-id-suspect` → **不**切 `blocked`,仅落 `onsite_discipline_log` + UI 琥珀 toast |
| 其他任意 0 命中(无明显 grep 语义) | 弱信号 | emit `discipline:trace-id-suspect` → 同上 |

#### Scenario: 主信号命中(AI 文本扫描)

- **GIVEN** 问题状态 `analyzing`
- **WHEN** Claude 输出的 assistant 文本含「这个 traceId 在三个目录里都未找到」
- **THEN** 1 秒内,问题状态变为 `blocked`
- **AND** 阻塞卡片 reason 注明触发源为"AI 文本扫描",matchedText 字段 = "未找到"

#### Scenario: 强信号命中(grep 家族扩展)

- **GIVEN** 问题状态 `analyzing`,traceId = `a1b2c3d4e5f6`
- **WHEN** Claude 调用 `Bash({ command: "rg 'a1b2c3d4e5f6' pod-1/" })` + 0 命中
- **THEN** 状态切 `blocked`,reason 注明触发源为"工具命令"

#### Scenario: 弱信号不自动 blocked

- **GIVEN** 问题状态 `analyzing`
- **WHEN** Claude 调用 `Bash({ command: "cat empty.log" })` 返回空内容
- **THEN** 状态**保持** `analyzing`
- **AND** `onsite_discipline_log` 新增一行 `kind='trace_id_suspect'`,含 cmd + stdout preview
- **AND** UI 弹琥珀 toast"⚠️ Claude 跑了 0 命中操作(非 grep 家族),请人工确认"

#### Scenario: 主信号必须配合上下文

- **GIVEN** Claude 跑了 5 个 grep,均 0 命中
- **WHEN** Claude 的 assistant 总结语说"traceId 在 5 个目录都未找到,可能是日志被截断"
- **THEN** 主信号命中 → 切 `blocked`
- **AND** reason 注明"5 个 grep 全部 0 命中 + AI 总结含'未找到'"

- **AND NOT** 切 `blocked` if Claude 跑了 5 个 grep 均 0 但**没**在总结语里说"未找到"(强信号仍触发)

### REQ-8.6 envelope discipline flag(替代 XML 标签猜)

The system MUST augment every Claude `assistant` and `tool_result` message envelope with a `discipline` field carrying detection flags. The client MUST render UI badges based on these flags, **not** by parsing text content for `<softening>` / `<traceId>` tags.

```ts
interface MessageEnvelope {
  // ... 现有字段
  discipline?: {
    traceIdEmpty?: boolean;    // 主信号或强信号命中
    traceIdSuspect?: boolean;  // 弱信号命中
    softeningWords?: string[]; // 命中的软化词列表
    triggeredAt?: string;      // ISO8601
  };
}
```

#### Scenario: 客户端按 flag 渲染徽章

- **GIVEN** 后端发的 assistant 消息 envelope 含 `discipline: { traceIdEmpty: true, triggeredAt: "..." }`
- **WHEN** 客户端渲染
- **THEN** 头部 `StatusBadge` 旁多一个琥珀徽章"⛔ 触发 traceId 阻塞"
- **AND NOT** 客户端去解析消息 body 找"未找到"字面量

### REQ-8.7 chat 路径隔离

The discipline middleware MUST be attached **only** to `ws.kind === 'onsite'` WebSocket connections. The middleware attachment function MUST check `enabledFor(ws) === true` before any detection runs.

#### Scenario: chat 路径不受影响

- **GIVEN** 用户在主 chat 页面(`/`)开 session
- **WHEN** Claude 跑 `grep 'X' empty.log` + 0 命中
- **THEN** 主 chat 状态不变、消息流不变、不产生任何 `discipline:*` 事件
- **AND** `chat-regression-baseline.txt` 前后对比通过(详见 tasks.md §5.5)
